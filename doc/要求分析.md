本書は、[企画書.md](企画書.md)で定義した方針（フェーズ1: 手動入力、フェーズ2: 学生証画像認識）に基づき、具体的なシステム仕様を策定するために整理した要求分析書です。

---

## 1. 利用者（学生）の要求とユーザー体験（UX）

ジムの利用者がスムーズかつストレスなく利用できるための要求項目です。

### 1.1 入力負荷の最小化
* **最小限のステップ数**: 画面でのタップや入力の手間を極限まで減らす。
* **片手操作への配慮**: 荷物を持っている状態を想定し、片手で完結する操作UI。
* **入力支援**:
  * 学科・学年はマスタテーブル（`departments_master` / `department_classes`）から選択式で提供し、表記揺れを防止する。学年の選択肢は選択した学科の修業年限（`years_count`）に応じて自動的に絞り込まれる。
  * 学籍番号を入力した段階で、過去の入力履歴（`users_cache`）から氏名やクラスを自動補完する。
  * 教職員番号で入力した場合は学科・学年入力を省略する。

### 1.2 混雑対策（スループットの向上）
* **処理時間の要件**: **1人あたりの入力完了時間を5秒以内**とする（2回目以降のキャッシュ補完時）。初回入力でも30秒以内を目安とする。
* **チェックイン確認画面との整合**: キャッシュ補完後に内容確認画面を1ステップ挟む設計とするが、自動遷移やデフォルトフォーカスの最適化により5秒目標を満たすこと。
* **画面遷移の最適化**: 「入力完了」のメッセージは3秒で自動的に閉じ、即座に次の人が入力できる状態に戻ること。
* **統計・ランキングの可視化**: 利用者と管理者が利用時間・連続利用日数を簡単に確認できること。

## 2. 管理者の要求と運用管理

ジムを管理する側が、データを正確に収集し、効率的に活用するための要求項目です。

### 2.1 データの確認・集計要求
* **集計の自動化**: 紙の台帳をExcelに転記する手間をなくすため、自動的にデジタルデータ化する。
* **出力フォーマット**: 管理者が扱いやすいフォーマット（CSV）での一覧エクスポート機能。
* **リアルタイム性**: 現在のジム利用状況（チェックイン中の在室人数）がリアルタイムで把握できることが望ましい。チェックアウトレコードの有無、または15時間自動チェックアウトにより在室判定を行う。
* **多言語表示**: 利用者に英語メッセージを選択させ、国際的な利用環境にも対応できること。

### 2.2 例外・エラー対応
* **データの修正・削除機能（論理削除）**: 利用者が間違えて送信したデータを、管理者が後から修正・削除できる仕組み。削除は**論理削除**（`deleted_at` カラムにタイムスタンプを記録）とし、30日以内であればゴミ箱から復元可能とする。30日経過後に完全削除（物理削除）する。
* **一括操作の誤操作防止**: 一括削除・一括学年UP等の重大操作には確認ダイアログを必ず表示する。
* **手動フォールバック**: 何らかの理由でデジタル入力ができない場合のバックアップ運用ルールの確立。

## 3. システム・環境・非機能要件

システムを設置・運用する物理的・技術的な環境に関する要求項目です。

### 3.1 設置環境とハードウェア
* **端末の選定**: 常時設置可能なノートPC（Windows）。将来的にRaspberry Pi + Camera Module 3への移行を見据えた設計とする。
* **ネットワーク依存度**:
  * インターネット接続（Wi-Fi等）が不安定になる可能性を考慮。
  * 一時的にオフライン状態になっても、データをローカルに保存し、オンライン復帰時に同期できる「オフラインファースト」の動作が望ましい。

### 3.2 セキュリティとプライバシー
* **キオスクモード（端末ロック）**:
  * 設置端末では、利用者がブラウザを閉じたり、OSの設定を変更したりできないようにする。
  * **実装方法**: Google Chromeを以下のオプションで起動し、全画面キオスクモードで動作させる。
    ```
    chrome.exe --kiosk --disable-infobars --disable-session-crashed-bubble
               --noerrdialogs --start-fullscreen <URL>
    ```
  * Windowsの場合、スタートアップ登録またはタスクスケジューラでPC起動時に自動的にキオスクモードで起動するよう設定する。
  * キオスクモード終了のショートカット（例: `Alt+F4`）については、OS側で無効化するかパスワード付きのスクリーンセーバーで保護する。
* **個人情報の保護**:
  * 端末の画面上に、他人の利用履歴（誰がいつチェックインしたか等）が表示されたままにならないようにする。
  * 入力完了後は**3秒**で初期画面（待受画面）に自動遷移する。
  * 60秒間無操作の場合も待受画面に自動遷移する（入力途中の個人情報流出防止）。

### 3.3 パフォーマンス
* **想定利用規模**: 約800名（全学科・全学年）。月間利用ログ件数は数千件〜最大1万件程度を想定。
* **応答速度**: チェックイン登録APIの応答時間は通常2秒以内とする（通信環境良好時）。
* **インデックス設計**: 800名規模では現行インデックス（`idx_usage_logs_student_id`、`idx_usage_logs_checked_in_at`）で十分だが、論理削除導入後は `deleted_at IS NULL` を含むクエリが増加するため、`deleted_at` カラムへの部分インデックス追加を検討する。

## 4. 第2フェーズ（画像認識）に向けた先行要件

第2フェーズ（学生証をかざすだけの自動認識）をスムーズに導入するための考慮事項です。

### 4.1 学生証の仕様確認
* **学生証フォーマットの統一性**: 全学科・全学年で学生証のデザインや、学籍番号・氏名の記載位置が同一であること。
* **OCRに適した書体**: カメラで読み取る文字が、光の反射やブレに強いコントラストと大きさを持っていること。

### 4.2 認識失敗時の設計
* **シームレスな切り替え**: カメラでの読み取りがタイムアウト、またはエラーになった場合、利用者を待たせずに即座に第1フェーズの手動入力画面へ遷移させる。
* **共通バックエンド**: 手動入力データと画像認識データが、同一のデータベースに同じフォーマットで保存されるような共通API設計。

## 5. 確定済み事項と残課題

### 5.1 確定済み事項（旧「今後の検討アジェンダ」より解決）

| 項目 | 決定内容 |
| :--- | :--- |
| **学科・クラスのマスタ管理** | `departments_master`（学科）・`department_classes`（クラス）テーブルで管理。手入力は廃止し選択式に統一。 |
| **退室（チェックアウト）記録** | フェーズ1から実装。チェックイン後15時間経過した場合は自動チェックアウト。 |
| **データ保存先** | Supabase（PostgreSQL）に確定。ホスティングはVercel。 |
| **削除方式** | 論理削除＋ゴミ箱（30日間保持・復元可能）を採用。 |

### 5.2 残課題
1. **キオスクモード終了時の運用フロー**: 管理者がメンテナンスのためキオスクモードを解除する手順をオペレーションマニュアルとして別途整備する。
2. **`deleted_at` インデックスの要否**: 運用開始後、論理削除件数が増加した時点でクエリパフォーマンスを計測し、部分インデックスの追加を判断する。

# ジム利用記録システム 技術選定書

本システムの開発・運用にあたって採用する技術スタック、アーキテクチャ、およびその選定理由を記述します。

---

## 1. 技術スタック概要

| 区分 | 技術・サービス名 | 選定バージョン（目安） | 役割 |
| :--- | :--- | :--- | :--- |
| **フレームワーク** | **Next.js** (App Router) | v14+ / React v18+ | フロントエンドUI、APIルート（バックエンド処理）、ルーティング |
| **ホスティング** | **Vercel** | - | クラウドインフラ（静的配信 ＋ Serverless Functions） |
| **データベース** | **Supabase** (PostgreSQL) | - | クラウドデータベース（利用ログ、学生キャッシュの保存） |
| **認証 (Auth)** | **Basic認証** (Next.js Middleware) | - | 管理者ページ（`/admin`）へのアクセス制限（ブラウザ標準認証） |
| **OCRライブラリ** | Tesseract.js | v4.x | ブラウザ上での学生証テキスト認識（フェーズ2） |
| **ローカル一時保存** | **IndexedDB** (または LocalStorage) | ブラウザ標準 | オフライン時のログ一時保存（耐障害性） |
| **クライアントアプリ** | Google Chrome (キオスクモード) | 最新安定版 | 設置PCで全画面UIとして実行するためのブラウザ環境 |

---

## 2. 各技術の選定理由

### 2.1 Webフレームワーク（Next.js）
* **選定理由**:
  * **Vercelへのデプロイの親和性**: Vercelと完全に統合されており、パフォーマンスの最適化やサーバーレス関数の配置が自動で行われます。
  * **フロントエンド・バックエンドの一体化**: App Routerの API Routes や Server Actions を利用することで、フロントエンドとバックエンド（Supabaseとの通信や認証）を単一のコードベースでシームレスに記述できます。

### 2.2 ホスティングインフラ（Vercel）
* **選定理由**:
  * サーバーの構築・管理（OSアップデートやミドルウェアのセットアップ）が一切不要な**サーバーレスインフラ**です。
  * Git（GitHub等）と連携することで、コードプッシュと同時に自動でビルド・デプロイが行われ、保守運用コストが極めて低くなります。

### 2.3 クラウドデータベース（Supabase）
* **選定理由**:
  * **Firebase代替のオープンソースPaas**: PostgreSQLをベースとしており、信頼性が高く、リレーショナルなデータ構造をクリーンに扱えます。
  * **高速なAPI生成**: テーブルを作成するだけで自動的にセキュアなREST APIが生成され、Next.jsからのデータ操作（CRUD）が容易になります。
  * **無料枠の充実**: 今回のジム利用記録（月間数千件程度）の規模であれば、十分に無料枠の範囲内で運用可能です。

### 2.4 認証（Basic認証）
* **選定理由**:
  * **最小限の開発・管理コスト**: ログイン用UI（画面）、セッション管理、認証情報のDBテーブルなどを構築する必要が一切ありません。
  * **Next.js Middlewareによる簡単実装**: `middleware.ts` を用いて、`/admin` 以下のパスにアクセスした際にブラウザ標準のBasic認証ダイアログを表示させ、Vercelの環境変数に設定したID・パスワードと照合するだけでセキュアな領域を確保できます。

### 2.5 OCRライブラリ（Tesseract.js）
* **選定理由**:
  * 学生証の画像をクラウドに送信することなく、**設置PCのブラウザ内で完結してOCRを実行**できます。
  * 個人情報（顔写真等）をインターネット上に送信しないため、セキュリティ・個人情報保護の観点から極めて安全であり、かつVercelのサーバーレス実行時間制限（Timeout）の影響を受けません。

### 2.6 ローカル一時保存（IndexedDB）
* **選定理由**:
  * インターネット接続が一時的に切断された場合でもチェックイン処理を継続するため、ブラウザ内の大容量ローカルストレージである IndexedDB にログを一時保存（キャッシュ）します。
  * ネットワーク復旧時に、自動的にVercel/Supabaseへ同期リクエストを送る「オフラインファースト」の動作を実現します。
